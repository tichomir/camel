## Context

The CaMeL interpreter is the core execution engine of the system.  It receives
pseudo-Python code generated by the P-LLM (which has never seen tool return
values) and executes it step by step.  During execution it:

1. Maintains a **variable store** mapping names to `CaMeLValue` instances.
2. **Propagates capability metadata** (sources, readers) across every operation.
3. **Evaluates security policies** before each tool call.
4. Raises a structured error for any AST construct outside the supported subset.

The interpreter operates on a *restricted* Python subset, not full Python.
This restriction is intentional: it eliminates side-channel vectors
(exceptions, generators, comprehensions, `import`) and keeps the trusted
execution surface as small as possible.

---

## Decision

### 1. `UnsupportedSyntaxError` — structured exception

```python
@dataclass
class UnsupportedSyntaxError(Exception):
    node_type: str   # e.g. "While", "Lambda", "ListComp"
    lineno:    int   # 1-based, taken from ast.AST.lineno (0 if unavailable)
    message:   str   # human-readable description
```

Raised whenever the interpreter encounters an AST node that is not in the
supported grammar.  The `node_type` field is always `type(node).__name__`,
enabling precise test assertions without string-matching the full message.

The `lineno` is taken from `getattr(node, "lineno", 0)` to handle synthetic
nodes that carry no line information.

---

### 2. `ExecutionMode` enum

```python
class ExecutionMode(Enum):
    NORMAL = "normal"   # capabilities propagated only via data assignments
    STRICT = "strict"   # control-flow conditions/iterables also propagate caps
```

**NORMAL mode** (default): capabilities flow only when values are assigned or
combined.  A variable assigned inside an `if` body carries only the
capabilities of its own RHS expression, not those of the `if` test.

**STRICT mode** (side-channel mitigation, PRD §6.3): before executing an `if`
or `for` body the interpreter accumulates the test / iterable expression's
capabilities into a *context capabilities* value (`ctx_caps`).  Every variable
assignment within the body then has `ctx_caps` merged into its capability set.
This closes the channel where an adversary could learn a private boolean by
observing which branch executes.

### 3. Supported AST node types

#### 3.1 Statement nodes (appear at the top level of a module or block)

| AST node        | Syntax example            | CaMeLValue wrapping / behaviour |
|-----------------|---------------------------|----------------------------------|
| `ast.Assign`    | `x = expr`                | Evaluate RHS → `CaMeLValue`; store under each target name.  Tuple-unpacking targets are supported (§ 5.2). |
| `ast.AugAssign` | `x += expr`               | Load LHS from store, eval RHS, apply operator via `propagate_binary_op`, store result. |
| `ast.If`        | `if cond: … else: …`      | Eval test.  In STRICT mode: pass test caps as `ctx_caps` into both branches.  In NORMAL mode: `ctx_caps` is unchanged. |
| `ast.For`       | `for x in iterable: …`    | Eval iterable.  In STRICT mode: merge iterable caps into `ctx_caps` for body; loop target variable also carries iterable caps.  `else` clause is NOT supported (raises `UnsupportedSyntaxError`). |
| `ast.Expr`      | `call(…)`                 | Evaluate contained expression for side effects only; result is discarded. |

All other statement types (`While`, `FunctionDef`, `AsyncFunctionDef`,
`ClassDef`, `Try`, `With`, `AsyncWith`, `AsyncFor`, `Return`, `Delete`,
`Import`, `ImportFrom`, `Global`, `Nonlocal`, `Assert`, `Raise`, `Pass`,
`Break`, `Continue`) raise `UnsupportedSyntaxError`.

`ast.Pass` and `ast.Break`/`ast.Continue` — while theoretically safe — are
intentionally excluded in Sprint 8 scope.  They may be added in a later sprint
with explicit loop-flow semantics.

#### 3.2 Expression nodes

| AST node           | Syntax example             | CaMeLValue wrapping strategy |
|--------------------|----------------------------|-------------------------------|
| `ast.Constant`     | `42`, `"hello"`, `True`, `None` | `wrap(node.value, sources=frozenset({"User literal"}), readers=Public)` |
| `ast.Name` (Load)  | `x`                        | Look up in `_tools` / `_builtins` / `_store`.  If not found: raise `NameError`. |
| `ast.BinOp`        | `a + b`, `a * b`, `a % b`, `a ** b`, `a // b`, `a / b`, `a - b` | Eval both operands → `propagate_binary_op(left, right, python_result)` |
| `ast.UnaryOp`      | `-x`, `+x`, `not x`, `~x` | Eval operand → new `CaMeLValue` with same caps, applied operator on raw value (use `propagate_assignment(operand, python_result)`) |
| `ast.BoolOp`       | `a and b`, `a or b`        | Left-to-right fold via `propagate_binary_op` for capability union; Python short-circuit semantics for the raw value |
| `ast.Compare`      | `a == b`, `a < b`, `a in c`, `a not in c` | Fold all operands left-to-right with `propagate_binary_op`; raw result is the boolean outcome of the full comparison chain |
| `ast.Call`         | `f(a, kw=b)`               | See §4 (Call evaluation) |
| `ast.Attribute`    | `obj.attr`                 | Eval `obj` → `obj_cv`; synthesise a key `CaMeLValue` for the attribute name (`sources=frozenset({"CaMeL"}), readers=Public`); call `propagate_subscript(obj_cv, key_cv, getattr(obj_cv.raw, attr))` |
| `ast.Subscript`    | `x[key]`                   | Eval container and key → `propagate_subscript(container_cv, key_cv, container_cv.raw[key_cv.raw])` |
| `ast.List`         | `[a, b, c]`                | Eval all elements → `propagate_list_construction(elem_cvs, [e.raw for e in elem_cvs])` |
| `ast.Tuple`        | `(a, b)`                   | Same as `ast.List` but result raw value is a `tuple` |
| `ast.Dict`         | `{"k": v}`                 | Eval all keys and values → `propagate_dict_construction(key_cvs, val_cvs, python_dict)` |
| `ast.JoinedStr`    | `f"hello {name}"`          | See §3.3 (f-string evaluation) |

All other expression nodes (`Lambda`, `IfExp`, `Set`, `ListComp`, `DictComp`,
`SetComp`, `GeneratorExp`, `Await`, `Yield`, `YieldFrom`, `Starred`,
`NamedExpr`) raise `UnsupportedSyntaxError`.

`ast.Index` (deprecated in Python 3.9, present in 3.8 ASTs) is handled
transparently: if encountered as the subscript slice, unwrap it and evaluate
the inner `value` node.

#### 3.3 f-string (`ast.JoinedStr`) evaluation

An f-string like `f"Hello {name}, your total is {total:.2f}"` is represented
as `ast.JoinedStr(values=[ast.Constant("Hello "), ast.FormattedValue(name), …])`.

Evaluation:
1. Iterate over `node.values`.
2. For each `ast.Constant`: wrap as a user-literal `CaMeLValue`.
3. For each `ast.FormattedValue`: evaluate the inner `value` expression.
   The `conversion` and `format_spec` fields are applied to the raw value when
   computing the string result.
4. The final raw value is the fully-formatted Python string
   (`"".join(str(part.raw) …)` with appropriate conversion/format_spec applied).
5. Capability result: fold all part `CaMeLValue`s left-to-right using
   `propagate_binary_op` to union all sources and readers.

`ast.FormattedValue` nodes with a non-`None` `format_spec` that is itself a
`JoinedStr` are supported (recursive eval).  `ast.FormattedValue` nodes with
`conversion = ord('a')` (ascii) or `conversion = ord('r')` (repr) are
supported; `conversion = -1` (no conversion) is the default.

### 4. Call evaluation (`ast.Call`)

```
_eval_Call(node, ctx_caps) → CaMeLValue
```

#### 4.1 Callee resolution

| Callee AST shape  | Resolution |
|-------------------|------------|
| `ast.Name`        | Look up `node.func.id` in: `_tools` first; then `_builtins`; then `_store` (if raw value is callable). |
| `ast.Attribute`   | Eval `node.func.value` → `obj_cv`; `callable = getattr(obj_cv.raw, node.func.attr)`.  Treat as a builtin (wraps return value). |
| Other             | Raise `UnsupportedSyntaxError` — arbitrary callee expressions are not supported. |

For `ast.Name` resolution the lookup order matters:
- `_tools` takes precedence because tool calls must go through the policy engine.
- `_builtins` are checked next for safe Python built-ins.
- `_store` falls back for cases where a callable was injected as a variable (rare;
  treated as a builtin — no policy check, return value is wrapped).

#### 4.2 Argument evaluation

All positional (`node.args`) and keyword (`node.keywords`) arguments are
evaluated in order, producing `list[CaMeLValue]` and `dict[str, CaMeLValue]`.

`ast.Starred` arguments (`*args`) and double-starred keyword unpacking
(`**kwargs`) raise `UnsupportedSyntaxError` — the P-LLM is instructed to
generate explicit arguments.

#### 4.3 Tool call path (callee in `_tools`)

1. **Policy check** (if `_policy_engine` is set): call
   `_policy_engine.check(tool_name, kwargs_mapping)` where `kwargs_mapping`
   maps each argument name to its `CaMeLValue`.  Positional args are mapped to
   parameter names using the tool's signature.
   - If the policy returns `Denied(reason)`: raise `PolicyViolationError`.
   - If the policy returns `Allowed()`: proceed.
2. **Call the tool** with raw values:
   `result_cv = tool(*[a.raw for a in pos_args], **{k: v.raw for k, v in kw_args.items()})`
3. **Type assertion**: the result MUST be a `CaMeLValue`.  If not: raise
   `TypeError(f"Tool {tool_name!r} must return CaMeLValue, got {type(result_cv)}")`.

#### 4.4 Builtin call path (callee in `_builtins` or attribute access)

1. Call with raw values:
   `result_raw = builtin(*[a.raw for a in pos_args], **{k: v.raw ...})`
2. Wrap with the union capabilities of all arguments:
   - If there are no arguments: `wrap(result_raw)` (sources=frozenset(), readers=Public).
   - Otherwise: use `propagate_list_construction` on all arg `CaMeLValue`s to
     obtain the union capability envelope, then construct a new `CaMeLValue`
     with `result_raw` as the value and the unioned sources/readers.

#### 4.5 Default builtins

The interpreter pre-registers these builtins by default (implementer may extend):

```
len, str, int, float, bool, list, dict, tuple, set, range,
sorted, reversed, enumerate, zip, map, filter, min, max, sum,
abs, round, isinstance, type, repr
```

`print` is intentionally **not** registered by default (side-effectful and a
potential side-channel for sensitive data).

### 5. Assignment semantics

#### 5.1 Simple assignment (`x = expr`)

```
_exec_Assign(node, ctx_caps)
```

1. Evaluate RHS → `rhs_cv`.
2. In STRICT mode: merge `ctx_caps` into `rhs_cv` (§ 6).
3. For each target in `node.targets`: call `_store_target(target, merged_cv)`.
4. `_store_target` dispatches on target type:
   - `ast.Name` (Store): `self._store[name] = value_cv`
   - `ast.Tuple` / `ast.List`: tuple-unpack (§ 5.2)
   - `ast.Subscript` / `ast.Attribute`: raise `UnsupportedSyntaxError`
     (in-place mutation of data structures is out of scope for Sprint 8)

#### 5.2 Tuple unpacking (`a, b = expr`)

Given a `ast.Tuple` or `ast.List` of `ast.Name` targets:
1. The container `CaMeLValue` has a raw sequence value.
2. For each `(i, name_target)` pair: extract element `i` using
   `propagate_subscript(container_cv, wrap(i, sources=frozenset({"CaMeL"})), container_cv.raw[i])`.
3. In STRICT mode: merge `ctx_caps` into each element.
4. Store each element under its name.

Nested unpacking (e.g. `(a, (b, c)) = …`) is **not** supported and raises
`UnsupportedSyntaxError`.

#### 5.3 Augmented assignment (`x += expr`)

1. Load current value of target from `_store` → `current_cv`.
2. Evaluate RHS → `rhs_cv`.
3. Apply the operator: `result_cv = propagate_binary_op(current_cv, rhs_cv, python_result)`.
4. In STRICT mode: merge `ctx_caps`.
5. Store under the target name.

Only `ast.Name` targets are supported for `AugAssign`.  `ast.Subscript` and
`ast.Attribute` targets raise `UnsupportedSyntaxError`.

### 6. STRICT mode and context capabilities (`ctx_caps`)

`ctx_caps: CaMeLValue | None` is threaded through all execution and evaluation
methods as a second argument after `node`.

**In NORMAL mode** `ctx_caps` is always `None` and `_merge_ctx_caps` is a
no-op.

**In STRICT mode** `ctx_caps` accumulates the capabilities of enclosing
control-flow conditions:

```
_merge_ctx_caps(value: CaMeLValue, ctx_caps: CaMeLValue | None) → CaMeLValue:
    if mode == NORMAL or ctx_caps is None:
        return value
    return propagate_binary_op(value, ctx_caps, value.raw)
```

When entering an `if` body:
```
test_cv   = _eval(node.test, ctx_caps)
inner_ctx = _merge_ctx_caps(test_cv, ctx_caps)   # union with outer ctx
exec body and orelse with inner_ctx
```

When entering a `for` body:
```
iter_cv   = _eval(node.iter, ctx_caps)
inner_ctx = _merge_ctx_caps(iter_cv, ctx_caps)
loop target variable also gets _merge_ctx_caps(element_cv, inner_ctx)
exec body with inner_ctx
```

The `ctx_caps` value used by the merge uses the **loop iterable** (not the
element), because which element we observe depends on iterating the iterable.

### 7. Session-state (variable store) lifecycle

```
self._store: dict[str, CaMeLValue]
```

| Event                  | Store behaviour |
|------------------------|-----------------|
| `CaMeLInterpreter()` constructed | `_store` is initialised to `{}` (empty). |
| `exec(code)` called    | Parses code, executes all statements.  New and updated variables are written into `_store`.  The store is NOT cleared between calls. |
| Multiple `exec()` calls | Each call sees the full store from all previous calls.  This enables sequential code blocks that reference variables set in earlier blocks. |
| Tool / builtin results  | Written into `_store` only when the call result is the RHS of an `ast.Assign`.  Results of bare `ast.Expr` calls are discarded. |

The store is pre-populated with **no values** at construction.  Callers that
want to seed variables (e.g. to inject a `CaMeLValue` from a prior step) may
call `interp.seed(name, value_cv)` — see §8.

### 8. `CaMeLInterpreter` public API

#### Constructor

```python
CaMeLInterpreter(
    tools:         Mapping[str, Callable[..., CaMeLValue]] | None = None,
    builtins:      Mapping[str, Callable[..., Any]]        | None = None,
    mode:          ExecutionMode = ExecutionMode.NORMAL,
    policy_engine: Any | None = None,
) -> None
```

| Parameter       | Description |
|-----------------|-------------|
| `tools`         | CaMeL-aware callables.  Each must return `CaMeLValue`.  Merged with (but override) the empty default dict. |
| `builtins`      | Plain Python callables.  Return value is wrapped by the interpreter.  Merged with the default builtin set. |
| `mode`          | `ExecutionMode.NORMAL` or `ExecutionMode.STRICT`. |
| `policy_engine` | Optional policy engine.  Must implement `check(tool_name: str, kwargs: Mapping[str, CaMeLValue]) -> PolicyResult`.  If `None`, all tool calls are allowed. |

#### `exec` method

```python
def exec(self, code: str) -> None:
```

Parses `code` via `ast.parse(code, mode="exec")` and executes every
top-level statement in order.  Raises:

- `UnsupportedSyntaxError` — if any AST node is not in the supported grammar.
- `PolicyViolationError` — if a policy engine is set and denies a tool call.
- `NameError` — if a `Name` node references an undefined variable.
- Any exception raised by a registered tool or builtin (propagated as-is).

#### `store` property

```python
@property
def store(self) -> dict[str, CaMeLValue]: ...
```

Returns a **shallow copy** of the current variable store.  Mutations to the
returned dict do not affect the interpreter's internal state.

#### `get` method

```python
def get(self, name: str) -> CaMeLValue: ...
```

Returns the `CaMeLValue` currently bound to `name`.  Raises `KeyError` if the
name is not in the store.

#### `seed` method

```python
def seed(self, name: str, value: CaMeLValue) -> None: ...
```

Injects a `CaMeLValue` into the store under `name`.  Useful for passing tool
outputs from a prior execution step into a new code block without re-executing.
Raises `TypeError` if `value` is not a `CaMeLValue`.

### 9. Unsupported construct list (negative test surface)

The following AST nodes MUST raise `UnsupportedSyntaxError` with the correct
`node_type` field.  This list forms the basis of the negative unit-test suite:

**Statements:**
`While`, `FunctionDef`, `AsyncFunctionDef`, `ClassDef`, `Try`,
`ExceptHandler`, `With`, `AsyncWith`, `AsyncFor`, `Delete`, `Import`,
`ImportFrom`, `Global`, `Nonlocal`, `Assert`, `Raise`, `Pass`,
`Break`, `Continue`, `Return`

**Expressions:**
`Lambda`, `IfExp`, `Set`, `ListComp`, `SetComp`, `DictComp`,
`GeneratorExp`, `Await`, `Yield`, `YieldFrom`, `Starred`, `NamedExpr`

> **Note on `Pass`, `Break`, `Continue`**: These are safe from a security
> perspective.  They are excluded in Sprint 8 to keep the scope minimal.
> They may be added in a subsequent sprint.

### 10. Operator mapping

The interpreter must map `ast.operator` / `ast.unaryop` / `ast.cmpop`
subclasses to Python operations applied to raw values:

#### Binary operators (`ast.BinOp`)

| AST operator class | Python operation |
|--------------------|-----------------|
| `ast.Add`          | `left + right`  |
| `ast.Sub`          | `left - right`  |
| `ast.Mult`         | `left * right`  |
| `ast.Div`          | `left / right`  |
| `ast.FloorDiv`     | `left // right` |
| `ast.Mod`          | `left % right`  |
| `ast.Pow`          | `left ** right` |
| `ast.BitAnd`       | `left & right`  |
| `ast.BitOr`        | `left \| right`  |
| `ast.BitXor`       | `left ^ right`  |
| `ast.LShift`       | `left << right` |
| `ast.RShift`       | `left >> right` |
| `ast.MatMult`      | `left @ right`  |

#### Unary operators (`ast.UnaryOp`)

| AST operator class | Python operation |
|--------------------|-----------------|
| `ast.UAdd`         | `+operand`      |
| `ast.USub`         | `-operand`      |
| `ast.Not`          | `not operand`   |
| `ast.Invert`       | `~operand`      |

#### Boolean operators (`ast.BoolOp`)

| AST operator class | Python semantics |
|--------------------|-----------------|
| `ast.And`          | Short-circuit `and` over raw values; union capabilities of ALL operands regardless of short-circuit result (conservative: evaluates all for capability tracking). |
| `ast.Or`           | Short-circuit `or` over raw values; union capabilities of ALL operands. |

**Why evaluate all operands for capability?** Short-circuit evaluation for
capabilities is unsound: an attacker could construct an expression like
`trusted_flag or injected_value` where Python would short-circuit and never
evaluate `injected_value`, but the capability union still tracks that
`injected_value`'s sources contribute to the branch decision.

#### Comparison operators (`ast.Compare`)

Each `ast.Compare` node has a `left` operand, a `comparators` list, and an
`ops` list.  Evaluation:

1. Eval `left` → `left_cv`.
2. For each `(op, comparator)` pair:
   - Eval `comparator` → `comp_cv`.
   - Fold capabilities: `accumulated = propagate_binary_op(accumulated, comp_cv, ...)`.
3. Compute the raw boolean using Python's chained comparison semantics.
4. Return `CaMeLValue(value=raw_bool, sources=…, readers=…)` from the fold.

Supported comparison operators: `Eq`, `NotEq`, `Lt`, `LtE`, `Gt`, `GtE`,
`Is`, `IsNot`, `In`, `NotIn`.

## Consequences

### Positive

- **Complete capability tracking.** Every expression site that produces a value
  is specified; there are no silent `Any` leaks from the interpreter.
- **Minimal attack surface.** The restricted grammar eliminates exception-based
  side channels, import-based injection, and generator-based lazy evaluation.
- **Composable.** `ExecutionMode.STRICT` can be enabled per-interpreter without
  changing any tool or policy code.
- **Independently testable.** Each `_eval_*` / `_exec_*` method has a clear
  contract that can be tested in isolation.

### Negative / Trade-offs

- **`AugAssign` on subscripts not supported.** `data["key"] += 1` requires
  in-place mutation semantics; deferred to a later sprint.
- **`for … else` not supported.** Rarely generated by P-LLM; not worth the
  implementation complexity.
- **Builtins wrapped conservatively.** All builtin argument capabilities are
  unioned; there is no per-builtin tracking of which arguments actually
  influence the output (e.g. `len(x)` carries all of `x`'s capabilities,
  even though `len` only observes the length).  This is sound (over-tainted)
  but may cause unnecessary policy denials in edge cases.

## Alternatives Considered

### A. Custom bytecode VM

Design a dedicated bytecode format and a CaMeL-specific virtual machine instead
of reusing Python's AST.

**Rejected because:**

- A custom bytecode format would require a complete toolchain: compiler,
  assembler, disassembler, debugger support.  The P-LLM would need to emit an
  unfamiliar, CaMeL-specific language, adding a non-trivial fine-tuning or
  prompting burden.
- Python's `ast` module is a mature, well-documented standard library component
  with stable node types across CPython releases.  It gives us syntax parsing
  for free; a custom VM would need to reinvent this.
- The restricted Python subset is instantly readable by engineers familiar with
  Python.  A bespoke bytecode is opaque without specialised tooling.

### B. Sandboxed full Python execution (e.g. `RestrictedPython`, `exec` in a limited `__builtins__` namespace)

Execute code in a restricted Python environment rather than walking the AST.

**Rejected because:**

- Full Python execution retains many side-channel vectors even with a stripped
  `__builtins__`: exception tracebacks reveal internal structure, generators
  introduce lazy evaluation, and `object.__subclasses__()` style escapes are
  well-documented.  Whitelisting every dangerous path is an ongoing arms race.
- Our AST walker explicitly *rejects* any node not in the supported grammar
  (`UnsupportedSyntaxError`).  This allowlist model is strictly safer than a
  blocklist approach, because unknown future Python constructs default to
  rejection rather than permission.
- Capability propagation must intercept *every* value-producing operation.
  In a full-execution model, intercepting `+` on two integers requires
  subclassing or `__add__` overrides on every Python type — fragile and
  incomplete.  In the AST walker, every `ast.BinOp` node passes through one
  `_eval_BinOp` method.

### C. Lua or JavaScript sandbox (embedded scripting language)

Embed a scripting language (Lua via `lupa`, or a JS engine via `quickjs` /
`pyduktape`) as the execution substrate.

**Rejected because:**

- The P-LLM would need to generate Lua/JS rather than Python.  Empirical
  evidence from the CaMeL paper shows state-of-the-art models generate
  Python most reliably.  Switching languages would degrade plan quality and
  increase retry rates.
- Foreign-language bridges introduce serialisation costs and type-mapping
  complexity.  `CaMeLValue` would need to be marshalled across the language
  boundary for every operation.
- Lua and JS sandboxes are not sandboxed by default; their own standard libraries
  require careful curation, similar to the problem with full Python execution.

### D. S-expression / Lisp-style DSL

Design a minimal expression language with explicit syntax for capability-tagged
values.

**Rejected because:**

- A Lisp-style syntax would be unfamiliar to the Python-centric LLM training
  corpus, reducing generation accuracy.
- The CaMeL paper authors' empirical evaluation was conducted with a Python
  pseudo-code format; deviating from this increases experimental uncertainty.
- A custom DSL requires parser maintenance.  Python's `ast.parse` is maintained
  by CPython.

### E. Haskell / Rust DSL (FW-1 future work)

Using a language with an algebraic type system and explicit error handling
(e.g. Haskell's `Either`, Rust's `Result`) would eliminate the exception
side-channel class entirely, because no implicit exception propagation exists.

**Deferred, not rejected.**  This is explicitly captured as open question FW-1
in the PRD.  For Milestone 1, Python AST is chosen for pragmatic delivery
reasons.  The Haskell/Rust DSL option remains the recommended long-term
direction once the security model matures.
