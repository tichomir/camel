## 1. Purpose & Scope

The CaMeL interpreter executes pseudo-Python code generated by the Privileged LLM
(P-LLM).  Although the P-LLM is trusted for *structure* (control flow, tool selection),
the code it generates runs in the same OS process as the CaMeL host application.
Without explicit namespace restrictions, adversarially crafted or mistaken P-LLM output
could:

- Import arbitrary standard-library or third-party modules to escape the intended
  execution sandbox (`import os`, `import subprocess`, `import socket`).
- Access dangerous builtins (`exec`, `eval`, `open`, `__import__`) to achieve
  arbitrary code execution or file-system access.
- Call timing primitives (`time.perf_counter`, `datetime.now`, `time.sleep`) to mount
  timing side-channel attacks that infer secret values from execution duration.

This document covers five sub-features that collectively close these vectors:

| ID | Description |
|---|---|
| M4-F10 | `ForbiddenImportError` — immediate raise on any `import`/`from … import` AST node |
| M4-F11 | Permitted builtin namespace — interpreter restricted to 16 approved names |
| M4-F12 | Timing primitive exclusion — `time`, `datetime`, and related names removed |
| M4-F13 | Central auditable configuration (`allowlist.yaml`) with mandatory review gate |
| M4-F14 | `ForbiddenNameError` — raised with offending name for any disallowed name access |

**Scope boundary:** This document covers the P-LLM code execution namespace.  It does
not address timing side channels arising inside *tool implementations* (those are the
responsibility of individual tool authors) or OS-level process timing channels (see §9).

## 2. Threat Model

### 2.1 Adversary Capabilities

CaMeL's threat model assumes an adversary who can:

1. **Control tool return values** — inject arbitrary string content into values returned
   by tools such as `read_email`, `fetch_url`, `read_document`.
2. **Observe the agent's external effects** — which tools are called, in what order,
   and (in some scenarios) wall-clock timing of the agent's responses.

The adversary cannot modify the P-LLM's system prompt, the user query, or the
platform security policies.  They also cannot observe the interpreter's internal
variable store directly.

### 2.2 Attack Classes Addressed by This Feature Cluster

#### Attack Class A — Arbitrary Code Execution via Module Import

If the P-LLM generates an `import os` statement (by mistake or via prompt injection),
the process's full file system access and subprocess capabilities are exposed.  Even if
the adversary cannot *write* the import statement directly, subtle injection patterns
could coerce the P-LLM into generating one.

**Mitigation:** M4-F10 intercepts `ast.Import` and `ast.ImportFrom` nodes *before any
module loading occurs*, raising `ForbiddenImportError` immediately.  M4-F11 also
excludes `__import__` and `importlib` from the permitted namespace, blocking indirect
import paths.

#### Attack Class B — Dangerous Builtin Abuse

Python builtins such as `exec`, `eval`, `open`, `compile`, and `__import__` can achieve
arbitrary code execution, file-system I/O, or dynamic code generation.  These must not
be available in the P-LLM execution namespace.

**Mitigation:** M4-F11 defines an explicit *allowlist* of safe builtins.  Any name not
on the allowlist raises `ForbiddenNameError` (M4-F14) before the name is resolved.

#### Attack Class C — Timing Side-Channel via Direct Primitives

An adversary who controls tool return values can craft content that causes the
interpreter to take secret-dependent branches (e.g., `if secret_token == "X": ...`).
If timing primitives are available, the adversary can measure execution duration via an
external clock and infer which branch was taken, recovering the secret value without
ever reading it directly.

**Mitigation:** M4-F12 excludes all timing-related names from the interpreter namespace
(see §5 for the full exclusion list and per-name rationale).  This closes the *direct
timing primitive* vector; residual indirect timing channels are documented in §9.

### 2.3 Non-Goals

- This feature cluster does not defend against adversary-controlled timing channels that
  do not involve explicit timing primitives (e.g., iteration-count channels).
- It does not prevent all ways a compromised P-LLM could behave maliciously — capability
  and policy enforcement layers provide the primary data-flow defence.
- It does not address timing channels inside tool implementations.

## 3. Allowlist Architecture Overview

The enforcement layer has three components, all active at interpreter startup and
execution time:

```
┌─────────────────────────────────────────────────────────┐
│               CaMeLInterpreter.__init__()                │
│                                                         │
│  1. AllowlistLoader.load("camel/config/allowlist.yaml") │
│     → AllowlistConfig (Pydantic-validated)              │
│                                                         │
│  2. build_permitted_namespace()                         │
│     → {name: builtin_fn, ...}  (16 entries)             │
│     → stored as self._permitted_namespace (immutable)   │
└─────────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│               CaMeLInterpreter.exec(code)               │
│                                                         │
│  AST parse → walk statements:                           │
│                                                         │
│  ┌─ ast.Import / ast.ImportFrom ──────────────────────┐ │
│  │  emit ForbiddenImportEvent to audit log             │ │
│  │  raise ForbiddenImportError(module_name, lineno)   │ │
│  └────────────────────────────────────────────────────┘ │
│                                                         │
│  ┌─ ast.Name (Load) ──────────────────────────────────┐ │
│  │  1. Check self._store → return CaMeLValue if found  │ │
│  │  2. Check self._permitted_namespace → wrap & return │ │
│  │  3. emit ForbiddenNameEvent to audit log            │ │
│  │     raise ForbiddenNameError(name, lineno)          │ │
│  └────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

The `AllowlistLoader` (in `camel/config/loader.py`) parses, validates, and caches the
`allowlist.yaml` configuration.  The permitted namespace is constructed once per
interpreter instance and stored as an immutable `dict`; it is never mutated after
`__init__` completes.

## 4. Permitted Builtin Rationale (M4-F11)

The permitted builtin set is intentionally minimal: 16 names covering the data
manipulation and control-flow primitives a P-LLM needs to write well-formed execution
plans.  Each name was individually reviewed against the following criteria:

1. **No I/O side effects** — must not read/write files, network, or standard I/O
   beyond the controlled `print()` display channel.
2. **No code generation or dynamic evaluation** — must not allow compilation or
   execution of new code.
3. **No timing observability** — must not expose wall-clock or CPU time.
4. **Bounded resource use** — must not enable unbounded memory or CPU consumption
   that could be exploited for side-channel attacks.

### 4.1 Approved Builtin Catalogue

#### Type Constructors

| Name | Rationale |
|---|---|
| `list` | Pure in-memory data structure constructor.  No I/O, no side effects.  Required for collecting tool results into sequences. |
| `dict` | Pure in-memory data structure constructor.  No I/O, no side effects.  Required for constructing keyword argument mappings. |
| `set` | Pure in-memory data structure constructor.  No I/O, no side effects.  Useful for deduplication within P-LLM plans. |
| `str` | String type and constructor.  No I/O, no side effects.  Required for explicit string conversion in tool arguments. |
| `int` | Integer type and constructor.  No I/O, no side effects.  Required for numeric argument construction. |
| `float` | Float type and constructor.  No I/O, no side effects.  Required for numeric argument construction with decimal precision. |
| `bool` | Boolean type and constructor.  No I/O, no side effects.  Required for conditional logic in P-LLM plans. |

**Excluded alternatives:** `bytes`, `bytearray`, `memoryview`, and `complex` are not
admitted — they are not needed by typical agent plans and introduce niche attack
surface (e.g., binary data manipulation or numeric side channels via complex arithmetic).

#### Type Checking

| Name | Rationale |
|---|---|
| `isinstance` | Read-only type introspection.  No I/O, no side effects.  Required for P-LLM plans that branch on the type of a tool return value (e.g., handling both `str` and `list` responses from a Q-LLM extraction). |

**Excluded alternatives:** `type()`, `issubclass()`, `id()`, and `vars()` are not
admitted.  `type()` and `issubclass()` are rarely needed in agent plans; `id()` could
be used to construct timing-correlated identifiers; `vars()` exposes internal object
state and could be used to probe the interpreter's namespace indirectly.

#### Sequence Utilities

| Name | Rationale |
|---|---|
| `len` | Returns the length of a sequence.  Read-only, no side effects.  Ubiquitous in agent plans for checking list sizes before iteration. |
| `range` | Generates an integer range for iteration.  No I/O, no side effects.  Required for `for i in range(n)` patterns. |
| `enumerate` | Wraps an iterable with index counters.  No I/O, no side effects.  Avoids the `for i in range(len(x))` anti-pattern. |
| `zip` | Aggregates iterables element-wise.  No I/O, no side effects.  Required for parallel iteration over multiple tool outputs. |
| `sorted` | Returns a sorted copy of an iterable.  Purely functional — does not mutate the input.  Required for deterministic ordering of tool outputs before processing. |
| `min` | Returns the minimum element of an iterable.  Purely functional, no side effects. |
| `max` | Returns the maximum element of an iterable.  Purely functional, no side effects. |

**Excluded alternatives:** `sum`, `abs`, `round`, `divmod`, `pow`, `hash` are not
admitted for this version.  `hash` is explicitly excluded because Python's hash
randomisation (PEP 456) introduces timing-correlated behaviour.  The others are
considered non-essential for typical agent plans; they can be added via the review gate
if a concrete use case arises.

#### Output

| Name | Rationale |
|---|---|
| `print` | Routed by the interpreter to a separate `DisplayChannel` (M2-F10).  Never written to the execution trace or forwarded to the P-LLM context.  Allows P-LLM plans to surface intermediate results to the user without security consequence.  The single exception to the "no I/O" rule, justified by the controlled routing. |

**Security note on `print`:** Although `print` performs I/O, the interpreter intercepts
it via the `DisplayChannel` abstraction.  The `print` builtin in `_permitted_namespace`
is replaced at construction time with the interpreter's internal display-channel
forwarder, not Python's raw `builtins.print`.  This ensures no output bypasses the
controlled channel.

## 5. Timing Primitive Exclusion Rationale (M4-F12)

### 5.1 Threat Scenario: Direct Timing Side Channel

Consider the following scenario:

1. The P-LLM generates a plan with a branch that depends on a private tool return value:
   ```python
   token = get_secret_token()   # sources: {"get_secret_token"}
   if token == "ABC123":
       result = process_a()
   else:
       result = process_b()
   ```

2. STRICT mode (M4-F2) correctly propagates `token`'s taint to `result`, so the policy
   engine will block any tool call that uses `result` unless explicitly authorised.

3. **Without timing primitive exclusion:** If `time.perf_counter` were available, an
   adversary could modify the plan to:
   ```python
   t0 = time.perf_counter()
   if token == "ABC123":
       time.sleep(1.0)   # secret-dependent delay
   t1 = time.perf_counter()
   duration = t1 - t0   # encodes the branch outcome
   exfiltrate(duration)
   ```
   Even though `exfiltrate(duration)` would be blocked by the policy engine (because
   `duration` carries the `token` source via STRICT mode propagation), the adversary
   can observe the agent's wall-clock response time externally, recovering the secret
   without triggering a policy violation on the *data path*.

4. **With timing primitive exclusion:** `time` and `perf_counter` are not in the
   permitted namespace.  The name lookup raises `ForbiddenNameError` before any timing
   measurement occurs.

### 5.2 Excluded Timing Name Catalogue

All names in this catalogue are absent from `permitted_builtins` and also listed in
`excluded_timing_names` within `allowlist.yaml` as an explicit documentation record.
The defence-in-depth step in `build_permitted_namespace()` also pops these names from
the constructed namespace even if a configuration error caused them to appear.

| Name | Module / Origin | Exclusion Rationale |
|---|---|---|
| `time` | `time` (stdlib module) | Primary namespace for all timing primitives.  Blocking the bare name prevents `time.sleep(...)` patterns even without an import statement. |
| `sleep` | `time.sleep` | Explicit delay primitive.  An adversary-controlled loop count can encode a secret value in total wall-clock duration. |
| `monotonic` | `time.monotonic` | High-resolution monotonic clock.  Directly usable for precision timing measurements. |
| `perf_counter` | `time.perf_counter` | Highest-resolution wall-clock timer in CPython.  Primary tool for precision timing attacks. |
| `perf_counter_ns` | `time.perf_counter_ns` | Nanosecond-resolution variant of `perf_counter`. |
| `process_time` | `time.process_time` | CPU-time measurement.  Useful for side-channel analysis independent of system load. |
| `time_ns` | `time.time_ns` | Nanosecond-resolution variant of `time.time`. |
| `clock` | `time.clock` (deprecated) | Removed in Python 3.8 but included for completeness against backport or monkey-patching scenarios. |
| `gmtime` | `time.gmtime` | Converts a timestamp to UTC struct_time.  Excluded to prevent timestamp-based covert channel construction. |
| `localtime` | `time.localtime` | Converts a timestamp to local struct_time. |
| `mktime` | `time.mktime` | Converts struct_time to a timestamp float (inverse of `localtime`). |
| `strftime` | `time.strftime` | Formats a timestamp as a string.  Excluded to prevent timestamp values from being embedded in tool arguments. |
| `strptime` | `time.strptime` | Parses a timestamp string.  Symmetric exclusion with `strftime`. |
| `datetime` | `datetime.datetime` | Provides `datetime.now()` and `datetime.utcnow()`, both direct timing primitives. |
| `timedelta` | `datetime.timedelta` | Represents a duration.  Could be used to compute elapsed time and leak secret-correlated branch counts. |
| `timezone` | `datetime.timezone` | UTC-offset tzinfo subclass.  Excluded to prevent timestamp-aware datetime construction in P-LLM code. |

### 5.3 Guidance for Legitimate Timestamp Use

If a CaMeL tool legitimately requires a timestamp (e.g., a calendar tool that needs to
record the current time), the tool *implementation* should obtain that timestamp from
`datetime.now()` internally and return it as a `CaMeLValue` with appropriate capability
annotations.  P-LLM-generated code must never obtain a timestamp directly.

## 6. Import Statement Interception (M4-F10)

### 6.1 AST-Level Blocking

The CaMeL interpreter's AST walker handles two statement node types by immediately
raising `ForbiddenImportError`:

| AST Node | Example | module_name extracted |
|---|---|---|
| `ast.Import` | `import os` | `node.names[0].name` → `"os"` |
| `ast.Import` (aliased) | `import sys as system` | `node.names[0].name` → `"sys"` |
| `ast.ImportFrom` | `from time import sleep` | `node.module` → `"time"` |
| `ast.ImportFrom` (future) | `from __future__ import annotations` | `node.module` → `"__future__"` |

The raise occurs **before any CPython module loading machinery is invoked**.  This is
a key property: it ensures no side-effecting module initialisation occurs even for
modules that have `__init__` code.

### 6.2 Exception Signature

```python
@dataclass
class ForbiddenImportError(Exception):
    """Raised when any import statement is found in interpreter-executed code (M4-F10).

    Attributes
    ----------
    module_name : str
        The module name extracted from the offending statement.
    lineno : int
        1-based source line number of the offending import statement.
    """

    module_name: str
    lineno: int

    def __str__(self) -> str:
        return (
            f"Import statements are forbidden in CaMeL-executed code: "
            f"'import {self.module_name}' at line {self.lineno} (M4-F10)"
        )
```

Defined in `camel/exceptions.py`, exported via `camel/__init__.py`.

### 6.3 Indirect Import Paths

| Attack vector | Mitigation |
|---|---|
| `__import__("os")` | `__import__` not in permitted namespace → `ForbiddenNameError` (M4-F14) |
| `importlib.import_module("os")` | `importlib` not importable (M4-F10) and not in permitted namespace |
| `builtins.__import__` | `__builtins__` attribute access not in permitted namespace → `ForbiddenNameError` |
| `exec("import os")` | `exec` not in permitted namespace → `ForbiddenNameError` |

### 6.4 Retry Loop Interaction

`ForbiddenImportError` is treated identically to `UnsupportedSyntaxError` by the
execution loop orchestrator: it is a **hard failure** that increments the outer retry
counter.  The module name string originates from P-LLM-generated code (not from tool
return values), so it is considered P-LLM-trusted content and is **not** subject to
exception message redaction (M4-F6).  The P-LLM regeneration prompt receives the
error type and line number.

## 7. ForbiddenNameError Enforcement (M4-F14)

### 7.1 Name Resolution Order

All `ast.Name` nodes in Load context pass through `_eval_Name`:

```
_eval_Name(node: ast.Name) -> CaMeLValue:

    name = node.id
    line = node.lineno

    # 1. Variable store — user-defined variables take precedence
    if name in self._store:
        return self._store[name]

    # 2. Permitted builtin namespace
    if name in self._permitted_namespace:
        raw = self._permitted_namespace[name]
        return wrap(raw, sources=frozenset({"CaMeL"}), readers=Public)

    # 3. Name not found in either scope — forbidden
    self._emit_allowlist_audit_event(
        event_type="ForbiddenNameAccess",
        offending_name=name,
        line_number=line,
        is_timing_primitive=(name in self._excluded_timing_names),
    )
    raise ForbiddenNameError(name, line)
```

Variable-store lookup is first for semantic correctness: a P-LLM variable named `len`
(unusual but valid) shadows the builtin consistently with Python semantics.

### 7.2 Exception Signature

```python
@dataclass
class ForbiddenNameError(NameError):
    """Raised when a name is not in the permitted interpreter namespace (M4-F14).

    Subclasses NameError so existing NameError handlers continue to work.

    Attributes
    ----------
    name : str
        The exact identifier from the ast.Name node.
    lineno : int
        1-based source line number of the name access.
    """

    name: str
    lineno: int

    def __str__(self) -> str:
        return (
            f"Name {self.name!r} is not permitted in CaMeL-executed code "
            f"at line {self.lineno} (M4-F14)"
        )
```

Defined in `camel/exceptions.py`, exported via `camel/__init__.py`.

## 8. Central Configuration & Review Gate (M4-F13)

### 8.1 Single Source of Truth

`camel/config/allowlist.yaml` is the **sole authoritative record** of the permitted
builtin set and the excluded timing names.  The file is:

- Checked into version control alongside the implementation code.
- Parsed and validated by `AllowlistLoader` (Pydantic model) at interpreter startup.
- Cached at module level by `load_allowlist()` (LRU cache, `maxsize=1`) so repeated
  interpreter construction does not re-read the file on the same process.
- Subject to the mandatory security review gate described in §8.2.

### 8.2 Security Review Gate Process

ANY modification to `allowlist.yaml` — additions, removals, or annotation changes —
**must** follow this process before merging:

1. **Open a pull request** describing the change, its security rationale, and which
   use case requires it.
2. **Reviewer approval:** A named reviewer from `review_gate.reviewers` (the security
   engineering team or a CaMeL core maintainer) approves with an explicit sign-off
   comment referencing the `allowlist.yaml` file.
3. **Inline rationale:** The rationale for the change is added as a YAML comment on
   the affected entry.
4. **Timestamp update:** `review_gate.last_reviewed` is updated to the review date
   (ISO-8601 format, e.g. `"2026-03-17"`).
5. **Risk annotation:** New entries carry a `risk_level` field (`low | medium | high`)
   and a one-line `justification`.

There is **no runtime override flag** that widens the permitted namespace.  If a
workflow requires a name not in the allowlist, the correct path is to register it as
a CaMeL tool with full capability and policy tracking — not to add it as a raw builtin.

### 8.3 Configuration Error Handling

The `AllowlistLoader` raises `ConfigurationSecurityError` (a `RuntimeError` subclass)
for any of the following conditions:

| Condition | Error |
|---|---|
| YAML file missing or unparseable | `ConfigurationSecurityError` |
| Pydantic schema validation failure | `ConfigurationSecurityError` |
| `review_required=true` but `last_reviewed` empty | `ConfigurationSecurityError` |
| `review_required=true` but `reviewers` list empty | `ConfigurationSecurityError` |
| Name appears in both `permitted_builtins` and `excluded_timing_names` | `ConfigurationSecurityError` |

These errors terminate interpreter construction before any P-LLM code is executed,
preventing the system from running in a misconfigured state.

## 9. Residual Risks

> **NG4 (PRD §3.2):** CaMeL does not guarantee side-channel immunity in all
> configurations.  Timing channels are *partially* mitigated, not eliminated.

The allowlist enforcement layer (M4-F10 – F14) closes the **direct timing-primitive
vector** but leaves the following residual risks explicitly out of scope:

| Risk | Description | Severity | Mitigation Status |
|---|---|---|---|
| **R1 — Indirect iteration-count channel** | Adversary-controlled loop count encodes a secret in total wall-clock execution duration without any explicit timing call.  Example: `for item in secret_list: expensive_op()` — the number of iterations (determined by `len(secret_list)`) is observable externally as execution time. | Medium | Partially addressed by STRICT mode M4-F1 (iterable's taint propagated to all loop-body assignments), which ensures the policy engine sees the taint before any downstream tool call.  Wall-clock observability itself is not mitigated. |
| **R2 — CPython instruction-dispatch variance** | CPython's interpreter dispatch time varies by expression complexity and branch taken.  A sufficiently precise external observer with a stable measurement channel could infer branch counts from micro-timing. | Low | Not mitigated.  Acknowledged in PRD §10 L3 and §7.3.  Mitigation would require constant-time interpreter implementation (future work, see PRD FW-1). |
| **R3 — Tool-implementation timing leakage** | Registered CaMeL tools may internally use timing-sensitive operations (e.g., database lookups proportional to secret size, network round-trips that vary with content). | Medium | Out of scope for the allowlist layer.  Tool implementers must apply constant-time techniques independently.  CaMeL provides no guarantees about tool implementation timing. |
| **R4 — OS-level process-time observation** | An OS-level attacker with access to `/proc/<pid>/stat` or equivalent can measure the agent process's CPU time, bypassing namespace exclusion entirely. | Low (requires OS access) | Out of scope.  Assumes the host environment provides process isolation.  Mitigation is an OS/deployment concern, not a CaMeL-layer concern. |
| **R5 — Future allowlist additions** | A future security review that incorrectly admits a timing-related name (e.g., `hash`, which uses randomised hashing with timing variance in CPython) could reintroduce timing channels. | Low (gated by review process) | Mitigated by the mandatory review gate (§8.2) and the `assert permitted ∩ excluded = ∅` invariant check at startup.  Risk increases if the review gate is bypassed. |
| **R6 — Deeply nested tool exception chains** | Exceptions propagating through multiple tool call frames before reaching the interpreter's exception handler may carry untrusted content from intermediary frames that are not captured in the immediate variable scope. | Low | M4-F6 dependency-graph-aware redaction addresses direct taint; deeply nested tool chains where intermediate frames have side effects remain a documented residual risk.  See `docs/design/milestone4-exception-hardening.md §9.2`. |

**Assessment:** Risks R1 and R3 are the most operationally significant.  R1 is partially
addressed by STRICT mode; R3 is a tool-layer responsibility.  The allowlist layer is a
necessary but not sufficient defence against all timing side channels.  It is designed
as one layer in a defence-in-depth stack alongside STRICT mode propagation (M4-F1–F4)
and the exception redaction engine (M4-F6–F9).

## 10. Rejected Builtins — Exclusion Decisions

The following Python builtins were reviewed and explicitly rejected for admission to
the permitted namespace.  This section provides the security rationale for each
exclusion decision.

### 10.1 Code Execution and Dynamic Evaluation

| Name | Exclusion Rationale |
|---|---|
| `exec` | Directly executes arbitrary Python code.  Would completely bypass the interpreter's AST-walking sandbox. |
| `eval` | Evaluates a Python expression string.  Same risk as `exec`; allows arbitrary expression evaluation outside the AST walker. |
| `compile` | Compiles Python source to code objects.  Combined with `exec`, allows full code injection. |
| `__import__` | Dynamic module import.  Bypasses the AST-level import ban (M4-F10). |

### 10.2 I/O and System Access

| Name | Exclusion Rationale |
|---|---|
| `open` | Opens files.  Direct file-system access outside the tool registry violates the sandbox boundary (NFR-1). |
| `input` | Reads from stdin.  Unexpected interactive I/O in a headless agent context. |
| `print` (raw) | The raw `builtins.print` is replaced at construction time by the interpreter's display-channel forwarder.  Admitting the raw builtin directly could bypass the display-channel routing. |

### 10.3 Introspection and Meta-Programming

| Name | Exclusion Rationale |
|---|---|
| `vars` | Returns the `__dict__` of an object or the current namespace.  Could expose interpreter internal state or be used to probe the variable store. |
| `dir` | Lists attributes of an object or the current scope.  Could expose interpreter internals and facilitate namespace exploration attacks. |
| `globals` | Returns the global namespace dict.  Direct access to the execution context. |
| `locals` | Returns the local namespace dict.  Same risk as `globals`. |
| `getattr` | Attribute access with dynamic name resolution.  Combined with known object handles, could be used to escalate access beyond the permitted namespace. |
| `setattr` | Attribute mutation.  Could modify internal state of objects in the interpreter's scope. |
| `delattr` | Attribute deletion.  Same risk as `setattr`. |
| `hasattr` | Probes attribute existence.  Lower risk than `getattr` but excluded to minimise introspection surface. |
| `type` | Returns the type of an object.  `isinstance` is admitted as the safer alternative for type-checking patterns. |
| `id` | Returns the object identity (memory address).  Could be used to construct timing-correlated identifiers or probe memory layout. |
| `callable` | Checks if an object is callable.  Not needed in typical agent plans; excluded to reduce introspection surface. |

### 10.4 Resource Management

| Name | Exclusion Rationale |
|---|---|
| `super` | Not useful without `class` definitions (which are also blocked). |
| `classmethod`, `staticmethod`, `property` | Decorators for class definitions; inapplicable since `class` is unsupported. |
| `object` | Base class access.  Combined with `type` and `getattr`, could enable metaclass manipulation. |
| `slice` | Slice object constructor.  Not needed; slice syntax is available directly. |
| `bytearray`, `bytes`, `memoryview` | Binary data manipulation.  Not required for typical agent plans; niche attack surface. |
| `frozenset` | Could be added if use cases arise.  Not needed for initial scope. |
| `complex` | Complex number arithmetic.  Not required for agent plans; numeric side channels via complex arithmetic are a theoretical risk. |

### 10.5 Exception Infrastructure

| Name | Exclusion Rationale |
|---|---|
| `Exception`, `ValueError`, `TypeError`, etc. | Exception *classes* are not needed for P-LLM plans, which do not contain `try`/`except`/`raise` (those constructs are blocked at the AST level). |
| `raise` (statement) | Blocked at the AST level as an unsupported construct. |

### 10.6 Conditionally Revisable

The following builtins are not admitted in this version but could be added via the
review gate if concrete, low-risk use cases arise:

| Name | Current Status | Conditions for Admission |
|---|---|---|
| `sum` | Not admitted | Low risk; admit if P-LLM plans frequently need numeric aggregation. |
| `abs` | Not admitted | Low risk; admit if P-LLM plans need absolute value. |
| `round` | Not admitted | Low risk; admit if P-LLM plans need floating-point rounding. |
| `any` | Not admitted | Low risk; admit if P-LLM plans need short-circuit existence checks. |
| `all` | Not admitted | Low risk; admit if P-LLM plans need universal quantification. |
| `map`, `filter` | Not admitted | Higher risk of opaque iteration; prefer explicit for-loops for auditability. |
